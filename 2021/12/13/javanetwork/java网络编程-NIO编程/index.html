五种网络IO模型

底层网络IO的实现方式

第一阶段：等待数据到达Socket缓冲区（操作系统为每个socket建立的一片内存系统区）。 wait for data，
第二阶段：从内核复制数据到用户程序区，这时用户才可以使用数据。 copy data from kernel to user

阻塞IO

非阻塞IO


第一阶段是非阻塞的，不断检查是否数据准备好，第二阶段阻塞读取数据
recvfrom -> [syscall -> wait ->] return no data ready
recvfrom -> [syscall -> wait ->] return no data ready
recvfrom -> [syscall -> wait ->] return ready
recvfrom -> [syscall -> copy ->] return OK

多路复用IO


每个IO都是非阻塞IO，第一阶段通过select/poll方法，一次性轮询多个IO句柄，检查是否有IO句柄准备好，第二阶段阻塞读取数据
select/pool -> [syscall -> wait ->] return readable
recvfrom -> [syscall -> copy ->] return OK

信号驱动IO


第一阶段构造一个信号处理器
第二阶段阻塞读取数据
signal handle -> [syscall -> wait ->] return
[syscall ->] signal handle -> recvfrom -> [syscall -> copy ->] return OK

异步IO


两阶段都是非阻塞
aio_read -> [syscall -> wait ->] return
[syscall -> copy ->] aio_read callback

五种模型的比较

简单NIO的使用

在BIO模式中，服务端是否有数据是靠客户线程去轮询，每一个客户都建立一个线程。在NIO模式中，不会为每个用户创建线程，假设有10K的连接数，程序的每一次循环中都需要去遍历10k的socket，检查socket中是否有数据可读，若10k个连接中只有1000个socket有数据，仍然需要遍历10k次，下一次循环又需要遍历10k次，但每次只有少部分有IO需求，浪费大量时间，为了解决该问题，操作系统提出了IO多路复用的技术，即IO的第三种模型。

多路复用IO技术的NIO模型

基本概念

1、Channel：ServerSocketChannel,SocketChannel（通道)

2、Selector：多路复用器（选择器)

3、Buffer：channel读写需要使用的缓冲区

Channel

  • 接口Channel：用于 I/O 操作的连接。
  • 抽象类SelectableChannel：可通过 Selector 实现多路复用的通道。为了与选择器一起使用，此类的实例必须首先通过 register 方法进行注册。此方法返回一个表示该通道已向选择器注册的新 SelectionKey 对象。可选择的通道要么处于阻塞 模式，要么处于非阻塞 模式。在阻塞模式中，每一个 I/O 操作完成之前都会阻塞在其通道上调用的其他 I/O 操作。在非阻塞模式中，永远不会阻塞 I/O 操作，并且传输的字节可能少于请求的数量，或者可能根本不传输字节。可通过调用可选择通道的 isBlocking 方法来确定其阻塞模式。
  • 新创建的可选择通道总是处于阻塞模式。在结合使用基于选择器的多路复用时，非阻塞模式是最有用的。向选择器注册某个通道前，必须将该通道置于非阻塞模式，并且在注销之前可能无法返回到阻塞模式。

SelectableChannel中的主要方法

  • abstract SelectableChannel configureBlocking(boolean block) ： 调整此通道的阻塞模式，true为阻塞，false为非阻塞。
  • SelectionKey register(Selector sel, int ops) ：向给定的选择器注册此通道，返回一个选择键。
  • abstract SelectionKey register(Selector sel, int ops, Object att) ： 向给定的选择器注册此通道，返回一个选择键。

ServerSocketChannel （监听)

ServerSocketChannel：A selectable channel for stream-oriented listening sockets。监听Sockets的SelectableChannel

1、具有阻塞和非阻塞两种模式

2、可以注册到多路复用器上

3、基于TCP连接

4、需要绑定到特定端口上

一、构造方法：
protected ServerSocketChannel(SelectorProvider provider) 受保护的方法，用户不能直接使用，只能通过该类的静态方法open()获取一个实例。
ServerSocketChannel ssc=ServerSocketChannel.open();

二、常用方法：
abstract SocketChannel accept() ： 接受到此通道套接字的连接，连接后产生一个SocketChannel。
static ServerSocketChannel open() ：通过调用此类的 open 方法创建服务器套接字通道。新创建的服务器套接字通道已打开，但尚未绑定。

abstract ServerSocket socket() ： 获取与此通道关联的服务器套接字，通过服务器套接字的bind()方法去绑定监听地址和端口。
SelectableChannel configureBlocking(boolean block) ：父类中的方法，调整此通道的阻塞模式。默认为true，阻塞模式。参数设置为false，将通道调整为非阻碍模式

SocketChannel（读写)

通过调用此类的某个 open 方法创建套接字通道。新创建的套接字通道已打开，但尚未连接。可通过调用套接字通道的 connect 方法连接该通道；一旦连接后，关闭套接字通道之前它会一直保持已连接状态。可单独地关闭套接字通道的输入端和输出端，而无需实际关闭该通道。调用关联套接字对象的 shutdownInput 方法来关闭某个通道的输入端将导致该通道上的后续读取操作返回 -1（指示流的末尾）。调用关联套接字对象的 shutdownOutput 方法来关闭通道的输出端

static SocketChannel open() 打开套接字通道。

static SocketChannel open(SocketAddress remote) 打开套接字通道并将其连接到远程地址。

abstract boolean connect(SocketAddress remote) 连接此通道的套接字。

long read(ByteBuffer[] dsts) 将字节序列从此通道读入给定的缓冲区。

long write(ByteBuffer[] srcs) 将字节序列从给定的缓冲区写入此通道。

abstract long write(ByteBuffer[] srcs, int offset, int length) 将字节序列从给定缓冲区的子序列写入此通道。
abstract SelectableChannel configureBlocking(boolean block) 调整此通道的阻塞模式。

SelectionKey register(Selector sel, int ops) 向给定的选择器注册此通道，返回一个选择键。

abstract SelectionKey register(Selector sel, int ops, Object att) 向给定的选择器注册此通道，返回一个选择键。
注意：采用channel读写数据要用缓冲区Buffer对象

Selector(多路复用器/选择器)

Selector类是Java提供的一个实现多路复用IO技术的类，它的底层调用了操作系统提供的IO复用技术，不同的操作系统，实现的复用技术不同。在应用过程中，用户可以根据并发数量的多少来创建多个Selector对象，一个通道可以注册到不同的selector中（要注意线程安全），但在同一个selector中，一个通道只能注册一次。

多路复用技术select、poll、epoll简介

  • 在详解Selector类之前，先讲解多路IO复用技术：
  • 多路复用技术：最早用在空管中，一个空管人员管理多架飞机的进出港及飞行状态，后来在通信领域大量使用多路复用技术提高通信能力。
  • 网络IO中的多路复用技术就是在单个线程中通过记录跟踪每一个Sock(I/O流)的状态来同时管理多个I/O流，使用该技术来提高服务器的吞吐能力。
  • 1983年，加州大学伯克利分校Unix开发小组就在BSD（Unix加州大学伯克利分校版）中引入了IO多路复用技术select的实现。

select 被实现以后，很快就暴露出了很多问题：

  1. 对socket进行扫描时是线性扫描，即采用轮询的方法，效率较低。
    当套接字比较多的时候，每次select()都要通过遍历FD_SETSIZE个Socket来完成调度，不管哪个Socket是活跃的，都遍历一遍，这会浪费很多CPU时间。
  2. select()仅仅告诉应用程序有几个sock(I/O stream)出现了数据，但是并不会告诉你是那个sock上有数据，还需调用于selectedKeys()方法去找出这些socket，由应用程序去遍历这些SelectionKey集合。
  3. select 只能监视1024个链接， linux 定义在头文件中的，参见FD_SETSIZE。
  4. 需要维护一个用来存放大量fd的数据结构，这样会使得用户空间和内核空间在传递该结构时复制开销大。
  5. select 不是线程安全的。
    select轮询方法对于早期的网络应用及PC服务器来说，还是能满足要求。

14年后，1997年，人们实现了poll, poll 修复了select的很多问题

  1. poll 去掉了1024个连接的限制，它的底层采用了链式结构。
  2. poll 从设计上来说，不再传入所有的socket的结构数组，传递的是一个pollfd结构的数组。
    3）poll还有一个特点是“水平触发”，如果报告了fd后，没有被处理，那么下次poll时会再次报告该fd。
    select和 poll系统调用的本质一样，管理多个描述符，也是进行轮循。包含大量文件描述符的数组被整体复制于用户态和内核的地址空间之间，而不论这些文件描述符是否就绪，它的开销随着文件描述符数量的增加而线性增大。

在2002年, Davide Libenzi 实现了epoll。
epoll 可以说是I/O 多路复用最新的一个实现，epoll 修复了poll 和select绝大部分问题：

  1. 没有最大并发连接的限制，能打开的FD的上限远大于1024（1G的内存上能监听约10万个端口）。
  2. 效率提升，不采用轮询的方式，而是事件响应方式（Reactor方式）。类似于Java中控件的响应方式，定义事件，定义事件的响应函数，一旦事件发生，则执行对应的函数，这个函数一般称为callback函数；即Epoll最大的优点就在于它只管你“活跃”的连接，而跟连接总数无关，因此在实际的网络环境中，Epoll的效率就会远远高于select和poll。
  3. 内存拷贝，利用mmap()文件映射内存加速与内核空间的消息传递；即epoll使用mmap减少复制开销。（共享内存的方式实现）

epoll支持水平触发和边缘触发，最大的特点在于边缘触发，它只告诉进程哪些fd刚刚变为就绪态，并且只会通知一次。还有一个特点是，epoll使用“事件”的就绪通知方式，通过epoll_ctl注册fd，一旦某个文件描述符就绪，内核就会采用类似callback的回调机制来激活该fd，epoll_wait便可以收到通知。（去掉了遍历文件描述符，而是通过监听回调的的机制，这正是epoll的魅力所在。）
目前，epoll只在linux里实现。因此，Java NIO的Selector(多路复用器)在不同的操作系统中有不同的实现。在Windows中任然采用poll轮询方式，而在linux中则采用epoll事件响应模式（Reactor）。在网络服务器操作系统的选择中，一般首选linux。
表面上看epoll的性能最好，但是在连接数少并且连接都十分活跃的情况下，select和poll的性能可能比epoll好，毕竟epoll的通知机制需要很多函数回调。

Selector的创建

Selector对象的创建是采用工厂方法，调用该类的open()方法完成。
Selector.open()方法创建一个Selector对象，在open方法中调用系统级默认的 SelectorProvider 对象的 openSelector 方法来创建的一个selector。SelectorProvider对象的实现会根据不同的操作系统选择不同的实现，Windows和Linux选择的复用技术是不同的，windows用的是poll而linux用的是epoll，因而效率也不一样。
Selector selector = Selector.open();

将SelectableChannel注册到selector中

注册Channel到Selector：
1、 selector中的channel必须是非阻塞
channel.configureBlocking(false);
2、将channel对象注册到指定的selector中，需要指明selector需要监听的IO操作，监听操作有4个：Connect、Accept、Read、Write，这4个操作用SelectionKey类的4个标志表示，SelectionKey.OP_CONNECT、SelectionKey.OP_ACCEPT、SelectionKey.OP_READ、SelectionKey.OP_WRITE，它们可以使用位或运算。比如若需要同时监听读和写，可以使用：
SelectionKey.OP_READ | SelectionKey.OP_WRITE 表示，实际应用中出现同时监听读又监听写的情况很少。对服务器来说，总是监听客户端的请求，有了请求后，在处理请求过程中就直接响应，这时的写操作没有必要去让Selector去监听。
服务器端有两个Channel，一个是ServerSocketChannel，它注册的事件是Accept。SocketChannel注册的是Read

register()方法的第二个参数。这是一个“interest集合”，意思是在通过Selector监听Channel时对什么事件感兴趣。可以监听四种不同类型的事件：
通道触发了一个事件意思是该事件已经就绪。所以，某个channel成功连接到另一个服务器称为“连接就绪”。一个ServerSocketChannel准备好接收新进入的连接称为“接收就绪”。一个有数据可读的通道可以说是“读就绪”。等待写数据的通道可以说是“写就绪”。
这四种事件用SelectionKey的四个常量来表示：
如果你对不止一种事件感兴趣，那么可以用“位或”操作符将常量连接起来，如下：
int interestSet = SelectionKey.OP_READ | SelectionKey.OP_WRITE;
第三个参数：可以在用register()方法向Selector注册Channel的时候附加对象

3、调用select()方法返回事件就绪的那些通道。
下面是select()方法：
int select()
int select(long timeout)
int selectNow()
select()一直阻塞到至少有一个通道在你注册的事件上就绪了。
select(long timeout)和select()一样，除了最长会阻塞timeout毫秒(参数)。
selectNow()不会阻塞，不管什么通道就绪都立刻返回。（此方法执行非阻塞的选择操作。如果自从前一次选择操作后，没有通道变成可选择的，则此方法直接返回零。）
select()方法返回的int值表示有多少通道已经就绪。亦即，自上次调用select()方法后有多少通道变成就绪状态。如果调用select()方法，有一个通道变成就绪状态，返回了1，若再次调用select()方法，如果另一个通道就绪了，它会再次返回1。如果对第一个就绪的channel没有做任何操作，现在就有两个就绪的通道。

4、通过调用selector的selectedKeys()方法，访问“已选择键集（selected key set）”中的就绪通道。Set selectedKeys = selector.selectedKeys();可以遍历这个已选择的键集合来访问就绪的通道。
public abstract class SelectionKey
{
public static final int OP_READ; public static final int OP_WRITE; public static final int OP_CONNECT;
public static final int OP_ACCEPT;
public abstract SelectableChannel channel(); public abstract Selector selector(); public abstract void cancel();
public abstract boolean isValid(); public abstract int interestOps(); public abstract void iterestOps(int ops);
public abstract int readyOps(); public final boolean isReadable(); public final boolean isWritable();
public final boolean isConnectable(); public final boolean isAcceptable();
public final Object attach(Object ob);
public final Object attachment();
}

Buffer的使用

Java BIO数据的读写通过字节数组就可以完成，而NIO中数据的读写操作则需要使用Buffer类来完成。Buffer的本意就是缓冲区的意思，它是一块可以写入数据，然后可以从中读取数据的内存，这块内存被包装成NIO Buffer对象，并提供了一组方法，用来方便的访问该块内存。
Buffer的直接子类有：ByteBuffer、CharBuffer、IntBuffer、LongBuffer、FloatBuffer…，我们这里主要使用ByteBuffer。
Buffer有三个基本属性：
capacity ：缓冲区的容量。
position： 读取或写入的元素的位置。
limit： 缓冲区有两个状态，一个是写状态，一个是读状态。当处于写状态时，limit表示能写的上限位置，它的值等于capacity。当转换成读状态时，limit表示缓冲区能读数据的上限位置。

缓冲区的操作：
clear()： 使缓冲区为读操作做好准备：它将限制设置为容量大小，将位置设置为 0。

flip()： 缓冲区读写状态的转换

static ByteBuffer allocate(int capacity) ：分配一个新的字节缓冲区。

static ByteBuffer wrap(byte[] array) ： 将 byte 数组包装到缓冲区中，并构造一个ByteBuffer 。

static ByteBuffer wrap(byte[] array, int offset, int length) ：将 byte 数组包装到缓冲区中，并构造一个ByteBuffer。

byte[] array() ：将ByteBuffer中的数据转换为 byte 数组。

get(byte[] dst) ：将ByteBuffer中的内容取出到dst。

put(byte[] src) ：将src中的内容放到ByteBuffer中。

put(byte[] src, int offset, int length)：将src中的内容放置到ByteBuffer中。

SelectableChannel 对象的多路复用器，也可称为选择器。它是Java NIO核心组件中的一个，用于检查一个或多个NIO Channel（通道）的状态是否处于可读、可写。如此可以实现单线程管理多个channels，也就是可以管理多个网络链接。
使用Selector的好处在于： 使用更少的线程就可以处理高并发，高吞吐。 相比BIO，一客户一线程的模式，大大减少了线程上下文切换带来的开销以及大量线程的开销。
