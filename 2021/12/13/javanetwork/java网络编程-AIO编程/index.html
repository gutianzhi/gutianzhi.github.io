java网络编程-AIO编程

发表于2021-12-13|更新于2021-12-13|java网络编程
|字数总计:1.7k|阅读时长:6分钟|阅读量:

异步和同步的区别：

POSIX的定义是这样子的：
A synchronous I/O operation causes the requesting process to be blocked until that I/O operation completes;

An asynchronous I/O operation does not cause the requesting process to be blocked;

两者的区别就在于synchronous IO做”IO operation”的时候会将process阻塞。按照这个定义，之前所述的blocking IO，non-blocking IO，IO multiplexing都属于同步IO。

有人会说，non-blocking IO不是叫非阻塞吗？这里有个非常“狡猾”的地方，定义中所指的”IO operation”是指真实的IO操作，就是例子中的recvfrom这个system call。non-blocking IO在执行recvfrom这个system call的时候，如果kernel的数据没有准备好，这时候不会block进程。但是，当kernel中数据准备好的时候，recvfrom会将数据从kernel拷贝到用户内存中，这个时候进程是被block了。

而asynchronous IO则不一样，当进程发起IO 操作之后，就直接返回再也不理睬了，直到kernel发送一个信号，告诉进程说IO完成。在这整个过程中，进程完全没有被block，业务逻辑将变成一个回调函数，等待IO操作完成后，由系统自动触发。也就是说业务的处理函数不是由应用程序去完成，而是由kernel去完成。AIO是Proactor(主动响应)
多路复用NIO是Reactor(事后响应)

和多路复用的Java NIO相比较，可以发现，异步IO是在数据读取或者写入调用已经完成的时候，再通知调用者，而非阻塞多路复用IO则是在有数据就绪，可以读写的时候通知调用者，读写仍然是由调用者执行，并且是阻塞的（这意味着如果要同时进行其他工作，要控制读写操作不能阻塞太长时间或者需要将其放去单独的IO线程执行)。

AIO

首先，每一个asynchronous channel都属于同一个group，共享一个Java线程池

  1. 从代码创建角度来看无论是AsynchronousServerSocketChannel还是AsynchronousSocketChannel的创建都使用各自的静态方法open，而open方法便需要AsynchronousChannelGroup。
  2. 可以发现AsynchronousChannelGroup其内部其实是一个(一些)线程池来进行实质工作的；而他们干的活就是等待IO事件，处理数据，分发各个注册的completion handlers。
    其次便是NIO与NIO2的区别了
    即REACTOR PROACTOR，同步IO和异步IO
    在同步IO中等待IO事件由我们注册的selector来完成，当感兴趣的事情来了，我们去读数据，处理数据。

在异步IO中等待IO事件同样为accept，read，write，connect，但读数据、处理数据交由系统完成，我们需要做的就是在事件响应接口中写好处理业务的代码。调用过程及什么时候调用的都与我们无关。（大家想一想UI编程中的按钮的编程过程）。
谁来干以前的活
是时候来想想NIO中（最朴素的），我们通常要干两件比较底层的事情

  1. 弄一个seletor，在它身上注册各种感兴趣的事件，然后select，等待感兴趣的事情发生
  2. 感兴趣的事情发生了，比如可以读了，这时便需要我们的线程从channel中读数据到bytebuffer里，并处理ByteBuffer中的数据。
    那么在AIO里，系统替你做的事是2，系统底层的AIO, windows下是iocp（真AIO），linux下是模拟epoll的工作方式，在NIO1中只响应IO就绪事件，在NIO2中，不仅响应就绪事件，还要调用事件接口中的方法去处理这些就绪事件。
    如何去发现就绪事件呢？NIO1中用Selector去发现，NIO2中就用AsynchronousChannelGroup去发现，因此在AIO中，产生的Channel由指定的一个group或默认的group来管理。

AsynchronousServerSocketChannel：

同BIO中的ServerSocket和NIO中的ServerSocketChannel，监听连接请求，接受连接请求。

1）、它的创建仍然使用该类提供的open()方法。

A）static AsynchronousServerSocketChannel open()
创建一个服务器channel， 并注册到默认group中

B）static AsynchronousServerSocketChannel open(AsynchronousChannelGroup group)
创建一个服务器channel， 并注册到指定的group中

2）、bind()方法：
abstract AsynchronousServerSocketChannel bind(SocketAddress local, int backlog)
绑定ip地址和端口号，指定请求队列的长度。不指定采用默认的长度

3）、accept()方法：

A）abstract Future accept()

接受一个连接请求，返回一个结果是AsynchronousSocketChannel的Future接口对象，用Future接口中的get()方法获得一个AsynchronousSocketChannel对象。用该对象可以去做读写事件的处理。用该方式编写的程序实际上等价于同步IO操作。一般不用该方式。

B）

接受一个连接请求，连接请求的处理交给事件响应接口中的completed方法去完成。它是异步处理的方式。一般使用该方式编写程序。

CompletionHandler接口（回调接口）

interface CompletionHandler<V,A>
类型参数：
V – 表示某事件执行成功后的结果类型
A – 表示处理某事件时可以传入的一个附件对象。（可以是一个协议对象，也可以是一个Buffer对象）
该接口中有两个方法需要实现：A）void completed(V result, A attachment)，当一个IO操作完成时调用该方法，方法中的result参数表示IO完成后的结果 。B）void failed(Throwable exc, A attachment) ，当一个IO操作完成失败时调用该方法。

AsynchronousSocketChannel

异步套接字通道，类同SocketChannel，不同点在于它可以异步完成读写操作。

异步读操作：

void read(ByteBuffer dst, A attachment, CompletionHandler<Integer,? super A> handler)
Reads a sequence of bytes from this channel into the given buffer.

异步写操作：

void write(ByteBuffer src, A attachment, CompletionHandler<Integer,? super A> handler)
Writes a sequence of bytes to this channel from the given buffer.
